Docker Compose는 단일 서버에서 여러 컨테이너를 관리하는 데 잘 동작해요. 하지만 서버가 10대, 100대로 늘어나면? 서버가 죽으면 누가 컨테이너를 다시 띄워주나요? 트래픽이 몰리면 자동으로 확장할 수는 없나요?

**Kubernetes(K8s)**는 이 문제를 해결하는 컨테이너 오케스트레이션 도구예요. 원하는 상태를 선언하면, Kubernetes가 그 상태를 유지해줍니다.

이번 주에는 먼저 **로컬에서 kind(Kubernetes IN Docker)**로 빠르게 실습한 뒤, EC2의 **k3s** 클러스터에 배포할 거예요.

### 2. Kubernetes 핵심 개념

#### 아키텍처

```
Control Plane (클러스터 관리):
  ├── API Server      ← 모든 통신의 중심 (kubectl도 여기에 요청)
  ├── etcd            ← 클러스터 상태 저장소 (Key-Value)
  ├── Scheduler       ← Pod를 어느 Node에 배치할지 결정
  └── Controller Manager ← 원하는 상태 ↔ 실제 상태 동기화

Worker Node (워크로드 실행):
  ├── kubelet         ← Pod를 실제로 관리하는 에이전트
  ├── kube-proxy      ← Service 네트워킹 (iptables/IPVS 규칙)
  └── Container Runtime (containerd)
```

#### 핵심 철학: 선언적(Declarative)

- **명령적:** "컨테이너를 3개 시작해라" → Docker Compose 방식
- **선언적:** "컨테이너 3개가 항상 실행 중이어야 한다" → Kubernetes 방식

YAML로 원하는 상태를 선언하면, Kubernetes가 현재 상태를 지속적으로 확인하며 맞춰줘요. 컨테이너가 죽으면 자동으로 다시 만들어요.

#### 핵심 오브젝트

**Pod:** 가장 작은 배포 단위. 보통 1개의 컨테이너 = 1개의 Pod. Pod는 언제든 죽을 수 있는 일회성 존재예요.

**Deployment:** Pod의 복제본을 관리. `replicas: 3`으로 설정하면 항상 3개를 유지. 롤링 업데이트와 롤백도 처리.

**Service:** Pod들에 대한 **안정적인 네트워크 엔드포인트.** Pod는 재시작될 때 IP가 바뀌지만, Service는 변하지 않는 DNS 이름과 IP를 제공해요.
- `ClusterIP`: 클러스터 내부에서만 접근 (기본값)
- `NodePort`: 모든 Node의 특정 포트로 외부 접근 (30000-32767)
- `LoadBalancer`: 클라우드 로드밸런서 자동 생성 (AWS에서는 ELB/NLB)

**ConfigMap / Secret:** 설정값과 민감한 정보를 Pod에 주입. 환경 변수 또는 파일로.

**Namespace:** 클러스터 내 논리적 격리. (default, kube-system 등)

## 프로젝트 실습 🎈

### Part A: 로컬에서 kind로 실습 (빠른 반복)

먼저 로컬에서 kind 클러스터를 만들어 빠르게 실험해요. SSH 없이 로컬에서 바로 `kubectl`을 쓸 수 있어서 디버깅이 훨씬 빠릅니다.

#### kind 설치

```bash
# kind 설치 (Mac)
brew install kind

# kind 설치 (Linux)
curl -Lo ./kind https://kind.sigs.k8s.io/dl/latest/kind-linux-amd64
chmod +x ./kind
sudo mv ./kind /usr/local/bin/kind

# kubectl 설치
# https://kubernetes.io/docs/tasks/tools/

# 클러스터 생성
kind create cluster --name study
kubectl cluster-info
kubectl get nodes
```

#### Docker Hub에 이미지 Push

kind는 로컬 Docker 이미지를 바로 쓸 수 없으므로, Docker Hub에 push하거나 `kind load`를 사용해요:

```bash
# 방법 1: Docker Hub
docker build -t <dockerhub-user>/todo-app:v1 .
docker push <dockerhub-user>/todo-app:v1

# 방법 2: kind에 직접 로드
kind load docker-image todo-app:v1 --name study
```

#### Deployment YAML 작성

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: todo-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: todo-app
  template:
    metadata:
      labels:
        app: todo-app
    spec:
      containers:
      - name: todo-app
        image: <dockerhub-user>/todo-app:v1
        ports:
        - containerPort: 3000
        env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: todo-secrets
              key: database-url
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 10
          periodSeconds: 5
        readinessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 5
          periodSeconds: 3
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 250m
            memory: 256Mi
```

#### 핵심 포인트 — 꼭 이해하세요!

- **Labels & Selectors:** Deployment의 `selector.matchLabels`와 Pod의 `labels`가 일치해야 해요. Service도 같은 labels로 Pod를 찾아요. K8s에서 모든 연결은 label 기반입니다.
- **Probes:**
  - `livenessProbe`: "이 컨테이너가 살아있나?" → 실패하면 컨테이너를 **재시작**
  - `readinessProbe`: "이 컨테이너가 요청을 받을 준비가 됐나?" → 실패하면 Service에서 **트래픽 제외** (죽이지는 않음)
  - Session 3에서 만든 `/health` 엔드포인트가 여기서 사용돼요!
- **Resources:** `requests` = 보장되는 최소치 (스케줄링에 사용), `limits` = 절대 초과 불가 (초과 시 OOMKill 또는 CPU 쓰로틀링)

#### Service YAML 작성

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: todo-service
spec:
  selector:
    app: todo-app
  ports:
  - port: 80
    targetPort: 3000
  type: ClusterIP
```

#### Secret 생성 및 배포

```bash
# Secret 생성 (DB 연결 문자열은 임시로)
kubectl create secret generic todo-secrets \
  --from-literal=database-url='postgres://todo:secret@todo-db:5432/tododb'

# 배포
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

# 상태 확인
kubectl get pods -w              # Pod 생성 과정 실시간 관찰
kubectl get svc
kubectl describe pod <pod-name>  # 상세 정보 + 이벤트 로그

# 로그 확인
kubectl logs <pod-name>

# 접속 테스트 (port-forward)
kubectl port-forward svc/todo-service 8080:80
# 다른 터미널에서:
curl http://localhost:8080/health
```

#### 로컬 실험 🔬

1. **Pod 삭제:** `kubectl delete pod <pod-name>` → Deployment가 즉시 새 Pod 생성! 이게 선언적 관리의 핵심.
2. **스케일링:** `kubectl scale deployment todo-app --replicas=5` → 5개로 확장. 다시 `--replicas=2`로 축소.
3. **잘못된 이미지:** 이미지 태그를 오타로 변경해보세요 → `ImagePullBackOff`. `kubectl describe pod`로 이벤트를 확인.
4. **셸 접속:** `kubectl exec -it <pod-name> -- sh` → 컨테이너 안에서 `curl`, `env` 등 실행.

### Part B: EC2에 k3s로 배포

로컬에서 충분히 실험했으니, 이제 EC2에서 실제 서비스를 노출해볼 거예요.

{{< alert context="warning" text="EC2에 직접 K8s 클러스터를 구성하는 것은 프로덕션에서는 일반적이지 않아요. AWS에서는 EKS 같은 매니지드 서비스를 사용하는 게 보통이지만, 학습과 비용을 고려해 k3s를 사용합니다." />}}

#### EC2 인스턴스 준비

새로운 EC2 인스턴스를 생성하세요:
- **OS:** Ubuntu 24.04 LTS
- **인스턴스 타입:** `t3.small` 이상 (2 CPU, 2GB RAM)
- **스토리지:** SSD 20GB 이상
- **Security Group:** 22(SSH), 80(HTTP), 443(HTTPS), 6443(K8s API), 31000(NodePort) 인바운드 허용

#### k3s 설치

```bash
# Docker 설치
curl -fsSL https://get.docker.com | sudo sh -

# k3s 설치
curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="server --tls-san=$(curl -s ifconfig.me) --write-kubeconfig-mode=644" sh
```

#### 로컬에서 원격 kubectl 접속 설정

```bash
# EC2에서 kubeconfig 확인
sudo cat /etc/rancher/k3s/k3s.yaml

# 로컬에 저장
# 내용을 복사해서 ~/.kube/remote-config 에 저장
# server 항목의 IP를 EC2 Public IP로 변경:
# server: https://[EC2-PUBLIC-IP]:6443

# 환경 변수 설정
export KUBECONFIG=~/.kube/remote-config

# 연결 확인
kubectl get nodes
```

#### Deployment & Service 배포

로컬에서 작성한 YAML을 그대로 사용하되, Service type을 **NodePort**로 변경하세요:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: todo-service
spec:
  selector:
    app: todo-app
  ports:
  - port: 80
    targetPort: 3000
    nodePort: 31000
  type: NodePort
```

```bash
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

# 외부에서 접속
curl http://[EC2-IP]:31000/health
curl http://[EC2-IP]:31000/api/todos
```

#### ALB 연결 (선택)

Session 6에서 만든 ALB의 Target Group에 EC2 인스턴스의 31000 포트를 추가하면, ALB를 통해서도 접속할 수 있어요.

### 디버깅 치트시트 📋

```
Pod가 안 뜨나요?
├── kubectl get pods              → 상태 확인
│   ├── Pending?                  → kubectl describe pod → 이벤트 확인
│   │    (리소스 부족? 이미지 pull 실패?)
│   ├── CrashLoopBackOff?         → kubectl logs <pod>  → 앱 에러 확인
│   │                               kubectl logs <pod> --previous → 이전 로그
│   └── Running인데 안 되나요?     → 아래로
│
├── kubectl get svc               → Service 확인
│   └── Endpoints가 비어있나?      → Labels 불일치!
│
└── kubectl exec -it <pod> -- sh  → 컨테이너 안에서 직접 확인
     ├── curl localhost:3000       → 앱이 내부에서 동작하나?
     ├── env                       → 환경 변수가 주입됐나?
     └── nslookup <service-name>   → DNS가 동작하나?
```
