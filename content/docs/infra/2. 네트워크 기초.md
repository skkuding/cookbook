## 공부할 내용 📚

### 1. IP 주소와 서브넷 (CIDR)

이 부분이 이번 주에서 가장 중요해요. AWS VPC 설계가 문자 그대로 "CIDR 범위 위에 서브넷을 나누는 것"이기 때문이에요.

#### IP 주소란?

- IP 주소는 네트워크에서 장치를 식별하는 **32비트 숫자**예요.
- `192.168.1.50`은 사실 `11000000.10101000.00000001.00110010`을 사람이 읽기 쉽게 쓴 것입니다.
- IPv4: 32비트 (약 43억 개), IPv6: 128비트 (사실상 무한)

#### CIDR 표기법

`/24`는 "앞에서 24비트가 네트워크 주소, 나머지가 호스트 주소"라는 뜻이에요.

| CIDR | IP 개수 | 사용 가능 | 특징 |
|------|---------|-----------|------|
| `/24` | 256 | 254 | 마지막 옥텟이 변함 |
| `/25` | 128 | 126 | |
| `/26` | 64 | 62 | |
| `/27` | 32 | 30 | |
| `/28` | 16 | 14 | |
| `/16` | 65,536 | 65,534 | AWS VPC 기본 크기 |
| `/8` | 16,777,216 | | |

**계산법:** 2^(32 - prefix) = 총 IP 수. 그리고 단계마다 반으로 줄어요.

#### Private IP vs Public IP (RFC 1918)

인터넷에서 직접 사용할 수 없는 **사설 IP 대역**이 있어요:

| 대역 | 범위 | 어디서 보게 되나 |
|------|------|------------|
| `10.0.0.0/8` | 10.0.0.0 ~ 10.255.255.255 | AWS VPC |
| `172.16.0.0/12` | 172.16.0.0 ~ 172.31.255.255 | Docker 기본 네트워크 |
| `192.168.0.0/16` | 192.168.0.0 ~ 192.168.255.255 | 가정용 공유기 |

이 대역의 IP는 인터넷에서 라우팅되지 않아요. 외부와 통신하려면 **NAT**가 필요합니다 (아래에서 배워요).

### 2. DNS — 이름을 IP로 바꾸는 시스템

브라우저에 `google.com`을 입력하면, 실제로는 `142.250.80.46` 같은 IP 주소로 접속해요. **DNS(Domain Name System)**가 이 변환을 담당합니다.

#### 동작 흐름

```
브라우저 → 로컬 캐시 확인 → OS 리졸버 (/etc/resolv.conf)
  → 재귀 리졸버 (8.8.8.8 등) → 루트 서버 → .com TLD → google.com 권한 서버
  → IP 주소 반환
```

#### 주요 레코드 타입

| 타입 | 설명 | 예시 |
|------|------|------|
| `A` | 이름 → IPv4 주소 | `google.com → 142.250.80.46` |
| `AAAA` | 이름 → IPv6 주소 | `google.com → 2607:f8b0:...` |
| `CNAME` | 이름 → 다른 이름 (별칭) | `www.example.com → example.com` |
| `NS` | 네임서버 지정 | `google.com NS ns1.google.com` |
| `MX` | 메일 서버 | 이메일 라우팅에 사용 |

**TTL (Time To Live):** DNS 응답이 캐시되는 시간. Google은 300초(5분)를 사용해요. TTL이 짧으면 변경 반영이 빠르고, 길면 DNS 쿼리 수가 줄어요.

{{< alert text="**나중에 만나게 될 곳:** AWS Route 53은 AWS의 DNS 서비스예요. EC2 인스턴스에 도메인을 연결할 때 A 레코드를, Load Balancer에 연결할 때 CNAME 또는 Alias 레코드를 설정하게 됩니다." />}}

### 3. 포트와 소켓

#### 포트란?

하나의 IP 주소에서 **여러 서비스를 구분**하기 위한 16비트 숫자 (0~65535)예요.

| 포트 | 서비스 | 언제 보게 되나 |
|------|--------|------------|
| 22 | SSH | 서버 접속 |
| 53 | DNS | 이름 → IP 변환 |
| 80 | HTTP | 웹 서버 |
| 443 | HTTPS | 암호화된 웹 서버 |
| 3000 | (관례) | Node.js 개발 서버 |
| 5432 | PostgreSQL | 데이터베이스 |
| 8080 | (관례) | 대체 HTTP 포트 |

**소켓 = IP + 포트 + 프로토콜.** `192.168.1.50:3000/TCP`는 하나의 고유한 네트워크 끝점이에요.

#### `0.0.0.0` vs `127.0.0.1` — 중요!

이 차이를 모르면 Docker와 Kubernetes에서 반드시 문제를 만나요.

- **`127.0.0.1` (loopback):** 자기 자신만 접속 가능. 외부에서 접근 불가.
- **`0.0.0.0` (모든 인터페이스):** 이 서버의 모든 네트워크 인터페이스에서 접근 가능.

Session 1 실험에서 `127.0.0.1`로 bind했을 때 외부에서 접속이 안 됐던 이유가 이거예요!

### 4. TCP와 UDP (간단하게)

우리가 만드는 웹 서버는 TCP를 사용해요. 깊이 들어가지 않지만, 차이점은 알아두세요.

| | TCP | UDP |
|---|-----|-----|
| 연결 방식 | 연결 수립 후 통신 (3-way handshake) | 연결 없이 바로 전송 |
| 신뢰성 | 순서 보장, 재전송 | 보장 안 함 |
| 속도 | 상대적으로 느림 | 빠름 |
| 용도 | HTTP, SSH, DB | DNS(쿼리), 영상 스트리밍, 게임 |

**한 줄 요약:** HTTP/HTTPS는 TCP 위에서 동작해요. DNS 쿼리는 보통 UDP, 응답이 크면 TCP로 폴백합니다.

### 5. NAT — 사설 IP가 인터넷에 나가는 방법

**문제:** 여러분의 서버가 `10.0.1.50` 같은 사설 IP를 가지고 있다면, 인터넷에서 이 IP로 직접 접근할 수 없어요.

**해결: NAT (Network Address Translation).** 라우터가 패킷의 출발지 IP를 자신의 공인 IP로 바꿔서 보내고, 응답이 오면 다시 원래 사설 IP로 돌려보내요.

```
내 서버 (10.0.1.50)
    → 라우터/NAT (출발지를 203.0.113.5로 변환)
        → 인터넷 (google.com)
        ← 응답은 203.0.113.5로 옴
    ← 라우터가 다시 10.0.1.50으로 변환
← 내 서버가 응답 수신
```

여러분 집의 공유기가 정확히 이 일을 하고 있어요!

{{< alert text="**AWS에서:** EC2 인스턴스가 private subnet에 있으면, 인터넷에 나가기 위해 **NAT Gateway**가 필요해요. Public subnet의 인스턴스는 **Internet Gateway**를 통해 공인 IP(Elastic IP)로 직접 나갈 수 있고요. Session 5에서 직접 구성해볼 거예요." />}}

### 6. 방화벽 — 들어오는 / 나가는 트래픽 제어

방화벽은 **규칙에 따라 트래픽을 허용하거나 차단**하는 장치예요.

#### 핵심 개념

- **인바운드 (Inbound):** 외부에서 내 서버로 **들어오는** 트래픽
- **아웃바운드 (Outbound):** 내 서버에서 외부로 **나가는** 트래픽
- **Stateful vs Stateless:**
  - **Stateful** (AWS Security Group): 인바운드를 허용하면, 그 연결의 응답은 자동으로 허용
  - **Stateless** (AWS NACL): 인바운드와 아웃바운드 규칙을 각각 설정해야 함
- **Default Deny:** 좋은 방화벽은 기본적으로 모든 것을 차단하고, 필요한 것만 허용해요.

예를 들어, 우리 서버에 필요한 규칙:
- 인바운드: SSH(22) 허용, HTTP(80) 허용, Node.js(3000) 허용
- 아웃바운드: 전체 허용 (서버가 외부 API를 호출할 수 있어야 하므로)

### 7. HTTPS와 TLS (개념만)

- **HTTP는 평문(Plaintext)이에요.** 중간에 누구든 내용을 볼 수 있어요.
- **HTTPS = HTTP + TLS.** 연결이 암호화되어 중간에 볼 수 없어요.
- **인증서:** 서버가 "나는 진짜 google.com이야"를 증명하는 문서. CA(Certificate Authority)가 발급.
- 브라우저의 자물쇠 아이콘 = TLS가 활성화되어 있다는 뜻.

{{< alert context="info" text="AWS에서는 ACM(AWS Certificate Manager)으로 무료 TLS 인증서를 발급받고, Load Balancer에 붙이는 방식으로 HTTPS를 설정해요. 개별 서버에서 인증서를 관리하지 않아도 됩니다." />}}

## 프로젝트 실습 🎈

이번 주 실습은 **실습 서버에서 직접 네트워크를 탐색하고 실험**하는 것이에요.

### 실습 A: 네트워크 탐색 (서버에서)

실습 서버에 SSH로 접속한 뒤, 아래 명령어들을 실행하고 결과를 이해해보세요.

```bash
# 1. 내 서버의 IP 주소와 네트워크 인터페이스 확인
ip addr show
# 질문: 인터페이스가 몇 개인가요? 각각의 IP는 무엇인가요?
# lo(loopback)와 eth0(또는 ens~)의 차이는?

# 2. 라우팅 테이블 확인
ip route show
# 질문: default 경로의 gateway IP는 무엇인가요? 이것이 NAT Gateway 역할을 합니다.

# 3. DNS 설정 확인
cat /etc/resolv.conf
# 질문: nameserver는 어디를 가리키고 있나요?

# 4. DNS 조회 실습
dig google.com A
dig google.com AAAA
dig +trace google.com        # 루트 서버부터 전체 해석 과정 추적!
nslookup google.com

# 5. 열린 포트 확인
ss -tlnp
# 질문: Session 1에서 만든 서버가 보이나요? 어떤 주소에 bind되어 있나요?

# 6. 외부 연결 추적
traceroute google.com        # 또는 tracepath
# 질문: 몇 개의 홉(hop)을 거치나요?
```

### 실습 B: CIDR 계산 연습

아래 문제를 풀어보세요. 계산기 없이 먼저 풀어보고, [CIDR Calculator](https://www.subnet-calculator.com/cidr.php)로 검증해보세요.

1. `10.0.0.0/24`에는 몇 개의 IP가 들어가나요?
2. `10.0.1.50`은 `10.0.1.0/26` 범위 안에 있나요? (`10.0.1.0` ~ `10.0.1.63`)
3. `10.0.1.100`은 `10.0.1.0/26` 범위 안에 있나요?
4. `10.0.0.0/16` 범위를 `/24` 서브넷으로 나누면 몇 개의 서브넷이 만들어지나요?
5. AWS VPC를 `10.0.0.0/16`으로 만들고, 아래처럼 서브넷을 나누려면 각각의 CIDR은?

```
VPC: 10.0.0.0/16
├── Public Subnet A:   10.0.1.0/?     (256개 IP)
├── Public Subnet B:   10.0.2.0/?     (256개 IP)
├── Private Subnet A:  10.0.10.0/?    (256개 IP)
└── Private Subnet B:  10.0.11.0/?    (256개 IP)
```

### 실습 C: 포트와 바인딩 실험

Session 1에서 만든 Node.js 서버를 수정하면서 실험해보세요.

```bash
# 1. 서버를 0.0.0.0:3000으로 실행 (기본)
node server.js

# 다른 터미널에서:
curl localhost:3000        # ✅ 동작
curl 127.0.0.1:3000       # ✅ 동작
curl [서버의실제IP]:3000   # ✅ 동작

# 2. 서버를 127.0.0.1:3000으로 변경하고 실행
# (server.js에서 listen('127.0.0.1', 3000) 으로 수정)
node server.js

curl localhost:3000        # ✅ 동작
curl [서버의실제IP]:3000   # ❌ 접속 안 됨!
# 왜? 127.0.0.1에만 bind했으므로 외부 인터페이스에서는 접근 불가
```

### 실습 D: 방화벽 (ufw) 실습

```bash
# ufw 상태 확인
sudo ufw status

# ufw 활성화 (주의: SSH 허용을 먼저!)
sudo ufw allow 22/tcp      # SSH 허용 — 이거 안 하면 서버에서 잠겨요!
sudo ufw allow 3000/tcp    # Node.js 서버 포트 허용
sudo ufw enable

# 상태 확인
sudo ufw status verbose

# 테스트: 3000번은 접속 되지만, 8080번은 안 되는 것 확인
curl [서버IP]:3000         # ✅
curl [서버IP]:8080         # ❌ (열려있는 서비스가 없거나, 방화벽에서 차단)

# 실험: 3000번 규칙 삭제 후 접속 시도
sudo ufw delete allow 3000/tcp
curl [서버IP]:3000         # ❌ 방화벽에서 차단!

# 원상복구
sudo ufw allow 3000/tcp
```

{{< alert context="danger" text="**주의:** `ufw enable` 전에 반드시 `ufw allow 22/tcp`를 먼저 실행하세요! 안 그러면 SSH 접속이 끊기고 서버에 들어갈 수 없게 돼요." />}}

```
IP/CIDR:    192.168.1.0/24 = 256개, /16 = 65,536개
```
