이번 주에는 인프라의 가장 기본이 되는 세 가지를 배워요. **Linux 서버를 다루는 법**, **HTTP 프로토콜의 동작 원리**, 그리고 **Node.js로 간단한 서버를 만드는 법**입니다.

프로덕션 서버에는 GUI가 없어요. 터미널만으로 서버를 다루는 것에 익숙해지는 것이 이번 주의 핵심 목표입니다.

## 서버 접속 가이드

팀에서 실습용 Ubuntu 서버를 제공합니다.
웹 UI도 같이 제공되니 그대로 쓰셔도 되고, 자기 컴퓨터에서 편하게 조작하고 싶다면 SSH를 사용해보세요.

### 접속 정보

```
호스트:    [팀에서 공유한 IP 주소]
포트:      22
사용자명:   [본인에게 부여된 계정명]
비밀번호:   [초기 비밀번호 — 접속 후 반드시 변경!]
```

SSH 접속 방법은 OS에 따라 다릅니다. Windows라면 WSL([설치 가이드](./Install%20WSL.md))이나 PowerShell의 OpenSSH를, Mac/Linux는 터미널에서 바로 `ssh username@서버IP`를 사용하세요.

{{< alert context="warning" text="비밀번호를 반드시 변경해주세요! 초기 비밀번호를 그대로 쓰면 다른 사람이 접속할 수 있어요." />}}

### (선택) SSH Key 인증 설정

비밀번호 대신 SSH Key로 접속하면 더 안전하고 편리해요. 관심 있다면 `ssh-keygen`과 `ssh-copy-id`를 검색해서 설정해보세요.

## 공부할 내용

### 1. Linux 기초

서버는 터미널로 관리해요. 아래 영역의 명령어들을 익히세요. 모두 외울 필요는 없고, 어떤 게 있는지 알아둔 뒤 필요할 때 찾아 쓰면 돼요.

- **파일 시스템 탐색:** 현재 위치 확인, 디렉토리 이동, 파일 내용 보기
- **파일/디렉토리 조작:** 생성, 복사, 이동, 삭제, 권한 변경
- **프로세스 관리:** 실행 중인 프로세스 확인, 종료, 백그라운드 실행
- **네트워크 도구:** `curl`로 HTTP 요청, `ss`로 열린 포트 확인
- **파이프라인:** `|`로 명령어를 연결하는 법 — `grep`, `sort`, `head`, `wc` 등과 조합

{{< alert text="**핵심 멘탈 모델:** Linux 철학은 '한 가지 일을 잘 하는 작은 도구를 만들고, 파이프로 조합한다'예요. 이 개념은 앞으로 계속 만나게 됩니다." />}}

- **텍스트 편집:** 간단한 오타 등을 고치기 위해서 원격 서버에서 파일을 가져와서, 편집하고, 다시 보내고... 하는 작업은 귀찮을 수 있어요. 터미널에서 직접 텍스트를 편집하는 `nano` 또는 `vi` 중 하나는 기본적으로 쓸 줄 알면 좋아요. 초보자에게는 `nano`를 추천해요.
- **패키지 관리:** `apt` 등 패키지 매니저로 소프트웨어 설치/삭제. 인터넷에서 재미있어보이는 패키지를 한번 다운받아보시기 바랍니다.

{{< alert text="**핵심 멘탈 모델:** Linux에서는 거의 모든 것이 파일이에요. 프로세스 정보(/proc), 장치(/dev), 네트워크 소켓까지 파일 시스템으로 표현됩니다. 이 개념은 나중에 Docker와 Kubernetes를 이해할 때 중요해져요." />}}

### 2. HTTP 프로토콜

HTTP는 웹에서 데이터를 주고받는 가장 기본적인 프로토콜이에요. 우리가 만들 서버는 HTTP로 통신합니다.

아래 질문에 답할 수 있을 때까지 공부하세요:

- HTTP의 요청-응답 모델이란? "무상태(Stateless)"는 무슨 뜻인가?
- HTTP 요청은 어떤 구조로 되어있나? (메서드, 경로, 헤더, 본문)
- GET, POST, PUT, DELETE의 용도 차이는?
- 200, 201, 301, 400, 401, 403, 404, 500 — 각각 언제 보게 되나?
- `Content-Type` 헤더는 왜 필요한가?

### 3. JavaScript & Node.js 기초

JavaScript는 웹 브라우저용 언어로 시작했지만, **Node.js** 덕분에 서버에서도 실행할 수 있게 되었어요. Node.js는 서버에서 JavaScript를 읽고 실행하는 런타임입니다.

특히 **비동기(Promise, async/await)** 개념은 반드시 이해하고 넘어가세요. Node.js의 핵심입니다.

### 실습: HTTP 서버 만들기

Express 없이, `http` 모듈만으로 서버를 만들어봐요.

**요구사항:**

1. `http` 모듈로 서버를 만드세요. 포트는 **3000번**으로 지정하세요.
2. Express를 쓰지 않고 `http` 모듈만으로 구현하세요. 파일 하나(`server.js`)면 충분해요!

#### `GET /`

정적 HTML 페이지를 반환합니다. `assets` 폴더에 있는 `week1.html` 파일을 `fs` 모듈로 읽어서 서빙하세요. [Github 링크](https://github.com/skkuding/cookbook/blob/main/content/docs/infra/assets/week1.html) (프론트용 자료)

- `Content-Type: text/html`로 응답해야 합니다.

#### `GET /api/info`

JSON으로 OS 정보를 반환합니다. `os` 모듈을 사용하세요.

```json
{
  "type": "Linux",
  "hostname": "infra-study-01",
  "cpu_count": 4,
  "total_memory_mb": 8192,
  "uptime_hours": 123.45
}
```

#### `GET /health`

서버가 살아있는지 확인하는 헬스체크 endpoint입니다.

```json
{
  "status": "ok"
}
```

이 endpoint는 사소해 보이지만, **모든 프로덕션 서비스에 존재해요.** 나중에 ALB의 Health Check(Session 5)와 Kubernetes의 liveness/readiness probe(Session 6)에서 이 endpoint를 사용하게 됩니다.

#### 그 외 경로

존재하지 않는 경로에 요청이 오면 `404` 상태 코드와 함께 에러 메시지를 반환하세요.

### 테스트

서버를 실행한 후, `curl`로 테스트해보세요. `-v` 옵션을 붙이면 HTTP 헤더까지 볼 수 있어요.

```bash
curl http://서버IP:3000/
curl http://서버IP:3000/api/info
curl http://서버IP:3000/health
curl http://서버IP:3000/nonexistent
curl -v http://서버IP:3000/api/info
```

### 실험해보기

아래 실험들을 직접 해보고, 왜 그런 결과가 나오는지 생각해보세요.

1. **서버를 종료한 뒤** `curl`을 보내보세요. 어떤 에러 메시지가 나오나요?
2. **같은 포트(3000)에 서버를 두 개** 실행해보세요. 어떤 에러가 나오나요?
3. `ss -tlnp`를 실행해서 3000번 포트에 어떤 프로세스가 붙어있는지 확인해보세요.
4. 서버 코드에서 listen 주소를 `127.0.0.1`로 바꿔보세요. 서버 내부에서는 접속이 되지만, **외부에서는 접속이 안 되는 것**을 확인해보세요. 왜 그런지 Session 2에서 배울 거예요.
